# App Document Skill

## 功能说明

将 `/app-analyze` 产出的分析结论，输出为标准化的两层×两格式文档体系：

- **Part1 代码实现层**：包含数据结构、函数地址、偏移量、资源路径等具体实现细节
- **Part2 方案说明层**：纯自然语言描述设计意图、技术选型、架构取舍，不依赖代码

每层各有两种格式：
- **MD 格式**：结构化关键词，AI 友好，低 token 消耗
- **HTML 格式**：暗色主题卡片布局，人类阅读友好，浏览器直接打开

## 使用方法

```
/app-document [--dir <输出目录>] [--topic <主题名>]
```

参数说明：
- `--dir`：可选，输出目录路径，默认为当前项目的 `decompile_output/`
- `--topic`：可选，主题名称用于文件命名，默认为 `analysis`

例如：
- `/app-document` — 基于当前会话的分析结论生成文档
- `/app-document --dir /path/to/output --topic terrain` — 指定输出目录和主题

## 设计来源

在 The Witness 分析中，用户明确提出了文档的两层分离需求：
1. "关于代码的实现部分，具体的结构，和重要的实现" → Part1
2. "基于实现的，形成的方案说明，说明是基于自然语言的，不借助代码来表达" → Part2

以及两种格式需求：
- "AI 方便阅读，少消耗 Token 的文件" → MD
- "一个 HTML 文件" → HTML

实践中还发现 HTML 文件过大会导致 Write 工具失败，需要用 Bash 分段追加写入。

## 部署方式：Subagent

本 skill 通过 Agent tool 启动 subagent 执行，原因：
- 文档生成是纯输出工作，输入是分析结论文本
- 不需要读取反编译文件，不需要主会话的搜索 context
- 隔离执行可以节省主会话 token

主会话触发后：
1. 收集当前会话中的分析结论（从最近的 `/app-analyze` 输出中提取）
2. 启动 subagent，注入分析结论和下方文档模板
3. subagent 生成 4 个文件后返回

subagent 配置：
- `subagent_type`: `general-purpose`
- `description`: `分析文档化 — <主题名>`

## 输出文件

```
<output_dir>/
├── part1_code_implementation.md      # 代码实现层 - AI友好
├── part1_code_implementation.html    # 代码实现层 - 阅读友好
├── part2_design_document.md          # 方案说明层 - AI友好
└── part2_design_document.html        # 方案说明层 - 阅读友好
```

## 执行流程

### Step 1: 收集分析结论

从当前会话上下文中提取 `/app-analyze` 的输出，包括：
- 核心数据结构
- 系统组成与子系统
- 处理流程
- 参数体系
- 资源组织
- 设计选型与取舍

### Step 2: 生成 Part1 代码实现层

#### Part1 MD 格式

结构模板：

```markdown
# <应用名> <主题>系统 — 代码实现拆解

> 来源: <反编译工具和版本>
> 函数总数: <N> | 符号状态: <正常/已strip>

## 1. 核心数据结构
### 1.1 <结构体名>
（伪代码格式的结构体定义，标注偏移量和字段含义）

## 2. 资源加载系统
### 2.1 Catalog 目录体系
（表格：Catalog名 | 资源类型 | 格式）
### 2.2 资源路径模式
（代码格式的路径模板）
### 2.3 关键加载函数
（表格：函数地址 | 功能 | 调用特征）

## 3. <主题相关的核心流程>
（流程步骤 + 关键函数引用）

## N. 资源打包结构
（表格：文件 | 内容 | 说明）
```

#### Part1 HTML 格式

- 暗色主题（背景 #0a0e17）
- 代码块使用语法高亮色（关键字红、类型蓝、注释灰、字符串青）
- 表格使用暗色边框
- 卡片式布局，每个章节一个 card

**HTML 分段写入规则**：
1. 先用 Write 工具写入 `<head>` + CSS + 页面头部
2. 用 Bash `cat >>` 追加每个章节的 HTML
3. 最后用 Bash `cat >>` 追加页面尾部

### Step 3: 生成 Part2 方案说明层

#### Part2 MD 格式

结构模板：

```markdown
# <应用名> <主题>系统 — 方案说明

> 纯自然语言描述，不依赖代码表达。

## 一、设计哲学
（为什么做这个选择，根本原因是什么）

## 二、<子系统A>方案
### 选型：<选择了什么>
（对比"采用"vs"未采用"，说明取舍）
（详细描述方案内容，用自然语言）

## 三、<子系统B>方案
...

## N、方案总结
| 维度 | 选型 | 核心取舍 |
（一句话总结整体设计思路）
```

**Part2 写作规则**：
- 不出现任何代码、函数名、偏移量、十六进制地址
- 不使用 `code` 标记包裹技术术语
- 用比喻和类比解释技术概念
- 每个子系统先说"选了什么"，再说"为什么这么选"，最后说"代价是什么"
- 流程描述用自然语言的步骤，不用伪代码

#### Part2 HTML 格式

- 同样暗色主题
- 使用 section-num 编号圆圈 + 渐变色区分章节
- "采用 vs 未采用"使用对比卡片布局（versus grid）
- 流程使用圆点步骤布局（flow-box + flow-step）
- 总结表格使用 summary-table 样式
- 结尾使用居中强调卡片

**HTML 分段写入规则**：同 Part1。

### Step 4: 输出汇总

```
文档生成完成：

- part1_code_implementation.md  — 代码实现层 (AI友好)
- part1_code_implementation.html — 代码实现层 (阅读友好)
- part2_design_document.md — 方案说明层 (AI友好)
- part2_design_document.html — 方案说明层 (阅读友好)
```

### 章节编号色板

每个章节使用不同的编号背景色，增加视觉区分度：
```
1: #1a3a2a (绿)  2: #2a1a3a (紫)  3: #3a2a1a (橙)

7: #3a1a2a (粉)  8: #1a2a3a (青)  9: #2a2a2a (灰)
```

## 注意事项

1. HTML 文件通常较大（几十KB），必须使用分段写入，不要尝试一次性 Write
2. Part1 和 Part2 的内容不应重复，Part1 是"怎么实现的"，Part2 是"为什么这么设计"
3. Part2 中绝对不要出现代码，如果需要提到技术概念，用自然语言描述
4. MD 格式追求信息密度，减少修饰语；HTML 格式追求阅读体验，可以适当展开
5. 如果分析结论中某些部分信息不足，在文档中标注"基于有限信息推测"而非编造细节
